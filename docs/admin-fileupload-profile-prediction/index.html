Design Proposal — AdminApp

個人タブ「新規登録 → ファイルアップロード」
読み取り → プロフ予測入力 詳細設計

履歴書・職務経歴書(PDF / Word / Excel / Markdown)をアップロードするだけで、AIが抽出した値を既存の登録フォームへ予測入力(下書き)として流し込み、管理者は確認・修正して保存するだけにする。入力工数を「全項目手入力」から「差分レビュー」へ。

1目的 — 手入力からレビューへ

入力UI: GenericRegistrationScreen(既存) 項目定義の正: engineer-profile-template.json 方式: initialData へのディープマージ注入

2現状の構成と発見した課題

現状は「アップロードして終わり」。Storage に保存し Firestore にメタデータを書く実装だが、コード調査で本番で機能しない・PIIが漏れるレベルの問題を確認した。

現状フロー(Agent ブランチ調査時点)

発見した不具合・リスク(B1〜B7)

B1 · P0 · バグ

💥 resume_uploads のルールが存在しない

firestore.rules に該当コレクションの定義がなくデフォルト deny。アップロード後のメタデータ setDoc は本番で permission-denied。UI上は失敗表示だが実ファイルは Storage に残る。

B2 · P0 · セキュリティ

🔓 履歴書PIIが全認証ユーザーに公開

match /uploads/{allPaths=**} が最上位で全域 read/write を許可。Storage ルールはどれか1つでも allow なら許可のため、下位の「本人+adminのみ」ルールが実質無効。個人・法人ユーザーも他人の履歴書を読み書き・上書きできる。

B3 · P1 · バグ

🔕 Web では Alert.alert が無反応

react-native-web の Alert.alert は no-op。AdminApp は Firebase Hosting 配信の Web アプリのため、エラーも成功通知も一切表示されない。

B4 · P1 · 不整合

📄 .txt が選べるのに拒否される

DocumentPicker は text/plain を許可するが ALLOWED_EXTENSIONS に txt がない。選択できてもバリデーションで弾かれる。

B5 · P1 · セキュリティ

🎭 拡張子のみの検証

マジックバイト(実体)検証がなく、.pdf にリネームした任意ファイルを受け入れる。Storage ルール側にも contentType / サイズ上限の強制がない。

B6 · P2 · UX

🗑️ 履歴がローカル state のみ

アップロード履歴はコンポーネント state に持つだけで、モーダルを閉じると消える。resume_uploads から購読すべき。

B7 · P2 · UX

🔗 キャンセル不可・紐付け導線なし

モーダルを閉じても uploadTask はキャンセルされない。userId: 'pending' を実ユーザーへ紐付ける導線が存在しない。

3あるべき全体アーキテクチャ

非同期・イベント駆動。クライアントはアップロードと Firestore 購読だけを行い、解析は Storage トリガーの専用ワーカーが担う。PII は GCP 内で完結させる。

① uploadBytesResumable ② メタデータ作成(status: uploaded) ③ finalize イベント ④ OCR(必要時) ⑤ 構造化抽出 ⑥ extractedProfile + status: parsed ⑦ onSnapshot 通知 ⑧ 保存(public_profile / private_info)+ applied 更新

設計判断(推奨と理由)

論点推奨理由
解析ワーカーの実行基盤Cloud Functions for Firebase v2
(Node20 / asia-northeast1 / onObjectFinalized)		実体は Cloud Run + Eventarc であり既存方針(Cloud Run 中心)と整合。トリガー配管・再試行・IAM を Firebase が面倒みる。常駐 API(admin node_backend)とはスケール特性が異なるため分離。
クライアント直解析
(ブラウザ内 pdf.js 等)		❌ 不採用API キー秘匿不可・端末性能依存・docx/xlsx 対応の重さ。監査ログもサーバ側に残せない。
構造化抽出 AIVertex AI Gemini(flash系)
responseSchema で JSON 強制		GCP 内で完結し PII を外部に出さない。スキーマ強制・低コスト。代替として Claude API(tool_use)も可だがデータ所在の観点で Vertex 優先。
OCRテキスト層のない PDF のみ
Document AI OCR		全件 OCR はコスト過大。まず pdf-parse でテキスト層を試す。
同期 / 非同期非同期(イベント駆動)
+ Firestore 購読で進捗通知		解析は 10〜60 秒かかり得る。HTTP 同期はタイムアウト・二重実行の温床。UI は status を追うだけ。

4Firebase 側の構成

🗂️

Storage レイアウト

uploads/resumes/
  ├ intake/{uploadId}/{fileName} — 新規流入(未紐付け)
  └ {userId}/{uploadId}/{fileName} — 紐付け後に移動

現行の pending_{timestamp} は uploadId と突合できず追跡不能。先に Firestore で docId を採番し、そのIDをパスに使う。intake は GCS ライフサイクルで90日自動削除。

⚙️

解析ワーカーの配置と冪等性

apps/admin_app/parse_worker/(TypeScript)を新設し、firebase.json に codebase: "resume-parser" で分離登録。トリガーは onObjectFinalized(1GiB / 300s / intake プレフィックス)。パスから uploadId を取り、トランザクションで uploaded → parsing に CAS 更新して二重イベントを排除。専用サービスアカウントに Storage 読取・Firestore 読書・aiplatform.user・documentai.apiUser の最小権限。App Check はクライアント SDK 側で段階導入。

resume_uploads スキーマと状態遷移

状態機械: uploaded → parsing → parsed → applied(失敗時 parse_failed → 再試行)
書き込み主体フィールド
クライアント(admin)
作成時		fileName fileType contentType fileSize storagePath uploadedBy uploadedAt status: uploaded
ワーカー
解析時		status parseStartedAt parsedAt parser(textExtractor / model / promptVersion) extractedProfile fieldConfidences warnings error retryCount — 抽出テキスト本文は保存しない(PII最小化)
クライアント(admin)
登録確定時		status: applied userId(採番された C…ID。'pending' 廃止) appliedAt appliedBy

セキュリティルール修正案

infrastructure/firebase/storage/storage.rules
// ⚠ 現行の match /uploads/{allPaths=**} { allow read, write: if request.auth != null; } は削除する。
//   (これが残る限り下位の制限ルールは無意味になる)
function isAdmin() {
  return request.auth != null && request.auth.token.role == 'admin';
}
function isValidResume() {
  return request.resource.size < 10 * 1024 * 1024
    && request.resource.contentType.matches(
         'application/pdf|application/vnd.openxmlformats-officedocument.(wordprocessingml.document|spreadsheetml.sheet)|text/(markdown|plain)');
}

// 新規流入: admin のみ。削除・上書き不可(create のみ)
match /uploads/resumes/intake/{uploadId}/{fileName} {
  allow read:   if isAdmin();
  allow create: if isAdmin() && isValidResume();
}

// ユーザー紐付け後: 本人 + admin のみ(書込はワーカー = Admin SDK でルール非適用)
match /uploads/resumes/{userId}/{allPaths=**} {
  allow read: if request.auth != null
    && (request.auth.uid == userId || isAdmin());
}

match /uploads/admin/{allPaths=**} {
  allow read, write: if isAdmin();
}

⚠ 他機能が uploads/ 直下の広域ルールに依存していないか、削除前に横断確認する(profile_images 等があれば個別ルール化して移行)。

infrastructure/firebase/firestore/firestore.rules — 追加(B1修正)
// 21. Resume Uploads (resume_uploads) — 管理者専用。ワーカーは Admin SDK なのでルール対象外
match /resume_uploads/{uploadId} {
  allow read: if isAdmin();
  allow create: if isAdmin()
    && request.resource.data.uploadedBy == request.auth.uid
    && request.resource.data.status == 'uploaded';
  // ステータス操作はワーカーと apply 処理に限定。クライアントからは applied 遷移のみ許可
  allow update: if isAdmin()
    && resource.data.status == 'parsed'
    && request.resource.data.status in ['applied']
    && request.resource.data.uploadedBy == resource.data.uploadedBy;
  allow delete: if isAdmin();
}

5「読み取り → プロフ予測入力」パイプライン

1
intake 検知 — onObjectFinalized(uploads/resumes/intake/ プレフィックス)
2
実体検証 — マジックバイト / サイズ再確認(拡張子偽装を排除)
3
テキスト抽出(形式別) — pdf-parse / mammoth / sheetjs / raw。スキャンPDFのみ Document AI OCR。先頭40,000字で打ち切り。抽出テキストはメモリ内のみ
4
LLM構造化抽出 — Vertex AI Gemini flash系。responseSchema で JSON 強制、温度0、JSONパース失敗時のみ1回リトライ
5
正規化・バリデーション — LLM出力を信用せずワーカー側で再検証(下表)
6
書き戻し — extractedProfile + fieldConfidences + warnings を resume_uploads へ(status: parsed)
7
クライアント通知 — onSnapshot で検知しレビューUIへ

テキスト抽出(形式別)

形式一次手段フォールバック
📕 PDFpdf-parse(テキスト層)抽出文字数 < 100字/頁 → スキャン判定 → Document AI OCR
📘 DOCXmammoth(表は Markdown 表に変換)—
📗 XLSXsheetjs(全シートをシート名付き TSV に直列化。スキルシート表対応)—
📝 MD / TXTUTF-8 読込(encoding-japanese で BOM / Shift_JIS 判定)—

LLM 構造化抽出 — スキーマはテンプレートから自動生成

🧬

二重管理の排除

抽出スキーマは engineer-profile-template.json からビルド時に自動生成(scripts/generate-resume-extraction-schema.js 新設)。テンプレート更新に抽出スキーマが自動追随する。

🎚️

displayType 別の生成ルール

文字列 → {value, confidence} / datePicker → YYYYMMDD / monthYearPicker → YYYYMM / singleSelectGroup・skillLevelSelect → 選択肢を enum 強制(スキル5段階はテンプレート内の実文字列が正)。

🚫

抽出対象外フィールド

パスワード・決済情報登録状況・繋がり情報登録状況・各種URL/アカウント項目はスキーマから除外。

🗣️

プロンプト方針(promptVersion 管理)

「記載がない項目は null(推測で埋めない)」「各項目に confidence 0.0–1.0」「習熟度は年数・役割の記述から最も近い選択肢を1つ」「職歴は新しい順に最大10件」。

正規化・バリデーション(ワーカー内)

項目ルール
メール / TELVALIDATION_RULES(registrationService)と同一の正規表現をワーカーへ移植。不一致は null + warning
生年月日YYYYMMDD 8桁整数化。和暦対応(昭和/平成/令和 → 西暦変換)
入学・卒業・入退社年月YYYYMM 6桁整数化
都道府県47都道府県の正規名へ(「東京」→「東京都」)
氏名姓/名分割。半角英名はスペース分割で Family/First。カナのみの場合は漢字欄 null
enum 系スキーマの enum 外の値は null に落として warning
confidence数値でない / 範囲外は 0 に丸め。null 項目はキーごと出力しない(マージ単純化)

エラー処理・運用

🔁 リトライ

OCR/LLM 失敗は parse_failed + error.code。UI から再試行(parsing CAS で再実行、retryCount ≤ 2)。スキーマ不適合は1回だけ再生成。

💰 コスト

flash系 + 平均5k字入力で 1件 ≒ 1円未満。Document AI は約 $1.5/1000頁(スキャンPDFのみ)。件数×単価をダッシュボード化。

🧾 監査

resume_uploads が監査ログを兼ねる — 誰が・いつ・何を・どのモデルで解析し・誰に適用したか。

6アプリ側の構成(expo_frontend)

新規レビューUIはほぼ作らない。既存の IndividualRegistrationModal(テンプレート駆動)に prefill を注入し、保存経路はテキスト入力フローと完全同一にする。

FileUploadModal のUI状態遷移(status を onSnapshot で追従)
🆔

ID事前採番(uploadResumeFile v2)

① Firestore で docId を先に採番し即 setDoc(ルール検証を先に失敗させる) → ② intake/{uploadId}/ にアップロード。sanitize(fileName): パス区切り・制御文字除去、255byte制限、NFC正規化。

🧩

prefill 注入

IndividualRegistrationModal に prefillData / prefillMeta prop を追加し、buildBlankInitialData() の結果に extractedProfile をディープマージ。テンプレートに無いキーは捨てる(スキーマ汚染防止)。

⚠️

信頼度の可視化

confidence < 0.7 の項目に警告色バッジ(共有側に fieldHighlights prop を小改修)。初期リリースは warnings 一覧をモーダル上部にまとめて表示するだけでも可(共有無改修)。

💾

保存経路は既存と同一

handleAdminUserSave → public_profile + private_info。新規保存ロジックは作らない。

🔗

applied 更新とファイル移動

保存完了時に resume_uploads を applied + 採番IDで更新。ワーカーが intake/ → uploads/resumes/{userId}/ へ移動し、private_info.resumeFiles[] に追記(本人・企業閲覧導線の起点)。

🔔

通知の置換

Web で無反応な Alert.alert を廃止し、クロスプラットフォームのトースト/ダイアログへ(B3)。
