02

🗺️ 現状マップ — いま何があるか

データは起動時に App.js が Firestore から一括取得し、DataContext 経由で5タブに配信。特権操作（BAN・一斉通知）だけ Node バックエンドを経由する構成。

図1: AdminApp の現状アーキテクチャ — 赤い破線 = 本番前に解消すべき箇所、橙 = 要改善

✅ 実装済み: 5タブの内訳

🏠

ホーム（KPI）

  • 選考ファネル（実データ）
  • ドリルダウン一覧
  • 満足度＝固定ダミー
  • 推移チャート＝固定ダミー
👤

個人

  • 検索 + 全体メール検索
  • スキルバッジ / ヒートマップ
  • 詳細・編集モーダル
  • BAN / 管理者メモ
🏢

法人

  • 検索・一覧・詳細閲覧
  • 招待コード発行
  • 権限の直接付与
  • 編集・審査は不可
📋

求人

  • 検索・一覧
  • スキル要件ヒートマップ
  • 詳細モーダル（閲覧）
  • 監視・非公開化なし
🎯

選考

  • 検索・一覧
  • 面接フィードバック編集
  • お見送り理由の表示
  • フィルタ未実装
03

🚧 ギャップ一覧 — 16件

優先度で絞り込めます。🔴 = 実装済みなのに未配線（最小工数で価値が出る）、🟠 = 要件文書に記載あり、🟡 = 本番運用の土台、⚪ = その他の改善候補。

📣

お知らせ一斉送信UIが未接続

NotificationSenderModal（298行・テスト付き）とバックエンドAPIは完成しているのに、どの画面からも import されていない。運営アナウンス配信の中核機能が眠っている状態。

🔴 未配線要件 4.2 / 4.21NotificationSenderModal.js工数: 小
🔔

通知ベルがモック（管理者向け通知なし）

ヘッダーのベルはバッジ件数が MOCK_DATA 固定でタップ不可。要件には「Admin側にも管理者向け通知を実装」とあるが、受信・一覧機能が丸ごと未実装。

🔴 未配線要件 4.2DashboardHelpers.js:29工数: 中
🎚️

クイックフィルタが全タブで無機能

「契約中」「募集中」「選考中」等のチップは全て console.log 止まり。繋がり推移の「個人×企業／個人×個人」切替も空実装（onPress={()=>{}}）。

🔴 未配線各タブ onApplyFilter工数: 小〜中
💰

成約管理（契約・請求・紹介料）

「契約書生成・管理（Admin側）」「紹介料計算」「リファンド管理（短期離職時の返金）」が存在しない。現状あるのは手数料率の設定のみ。

🟠 要件未実装要件 4.20【M24】工数: 大
🗂️

Admin設定 7カテゴリ中 4 が不足

未実装: ユーザー管理（ロール・権限・管理者アカウント）／招待管理（一覧・失効・使用状況）／規約更新管理／通知配信管理。実装済みはマッチング・手数料・メンテのみ。

🟠 要件未実装要件 4.21【M24】工数: 中〜大
🥧

サーベイ結果表示がダミー

「マッチング満足度」円グラフはハードコードの固定値。入社後サーベイ（1/3/6/12ヶ月）の集計表示が要件にあるが未実装。要件5.2「ダミーデータの使用禁止」にも抵触。

🟠 要件未実装要件 4.15 / 5.2OverviewTab.js工数: 中
📈

KPIチャートが固定配列

「登録ユーザー数推移」「繋がりの推移」も DashboardScreen 内のハードコード。設計書の改善提案にも createdAt ベースの実集計への移行が明記されている。

🟠 要件未実装設計書 改善提案 §1-2DashboardScreen.js:343工数: 中
🔐

管理者権限チェックが実質無効

バックエンド verifyAdmin の custom claims 検証がコメントアウトされており、有効なトークンなら誰でも BAN・一斉通知APIを実行可能。フロントも __DEV__ で admin 自動付与。最優先の是正項目。

🟡 本番土台設計書 認証・認可node_backend/index.ts:41工数: 小
📜

監査ログ（特権操作の履歴）なし

BAN・権限付与・設定変更・一斉通知などの特権操作の記録が残らない。設計書の admin_users には「操作ログ」と記載あり。管理ダッシュボードの定番要件。

🟡 本番土台設計書 admin_users工数: 中
🕵️

求人監視・モデレーションなし

設計書に JobMonitorScreen (Future)「不適切求人の監視」と記載。求人の非公開化・ステータス変更・通報対応のアクションが存在しない。

🟡 本番土台設計書 jd 連携工数: 中
🏛️

企業審査・法人プロフィール編集不可

権限マトリクスでは管理者は法人プロフィール「編集可（プルリク方式）」だが、CompanyDetailScreen は isEditable={false} の閲覧専用。設計書の「企業審査、アカウント管理」（承認/停止）も未実装。

🟡 本番土台要件 2.1 / 設計書CompanyDetailScreen.js:112工数: 中
🐘

スケーラビリティ / データ更新

起動時に全コレクションを一括 getDocs するのみ。pull-to-refresh・再取得・ページネーションなし。設計書も count() 集計やサーバーサイド集計への移行を提案。

🟡 本番土台設計書 改善提案 §3App.js fetchAllData工数: 中〜大
👯

設定画面の重複 + 保存先の不一致

App.js が使うのは features/platform_settings/ 版。features/settings/screens/ 版（250行・テスト付き）はデッドコード。保存先も設計書は platform_settings/global、実装は system_settings/matching_config と不一致 — メンテモードを他アプリが参照しているか要確認。

⚪ その他工数: 小
📲

プッシュ通知（FCM）未対応

通知は Firestore 書き込みのみで、ユーザー端末へのプッシュは飛ばない。バックエンドのコメントにも「将来のFCM等」と記載あり。

⚪ その他node_backend/index.ts:1工数: 中
📤

CSV / データエクスポートなし

ユーザー・企業・選考データの一括エクスポート機能がない。運営レポート作成や外部分析（スプレッドシート連携）の定番機能。期間指定つきKPIフィルタも将来候補。

⚪ 将来候補工数: 中
🗑️

退会・アカウント削除フローなし

BAN（無効化）はあるが、個人情報削除を伴う退会処理・データ削除リクエスト対応（プライバシー対応）の管理機能がない。問い合わせ（ヘルプ）対応の受け側管理も未実装。

⚪ 将来候補要件 5.1 関連工数: 中

05

🛣️ おすすめの着手順

「セキュリティ → 眠っている機能の解放 → 実データ化 → 管理機能の拡充」の順で、少ない工数から価値を積み上げる。

1

🔐 admin custom claims 検証の有効化 セキュリティ工数: 小

node_backend の verifyAdmin のコメントアウトを解除し、フロントの __DEV__ 自動付与を本番ビルドから排除。全ての前提となる是正。

2

📣 NotificationSenderModal の配線 未配線解放工数: 小

実装済みモーダルをヘッダーのベル or 設定画面から起動可能に。運営アナウンス配信（要件4.2）が即日使える状態になる。

3

🎚️ クイックフィルタの実装 未配線解放工数: 小〜中

「契約中」「募集中」「選考中」等のフィルタと繋がり推移のセグメント切替を実装し、日常運用の検索効率を上げる。

4

📊 ダミーチャート → 実データ集計 要件対応工数: 中

満足度（FMJSサーベイ）・登録推移（createdAt集計）を実データ化し、要件5.2「ダミーデータ禁止」を解消。KPIが信頼できる数字になる。

5

🗂️ 招待コード管理一覧 + 監査ログ 管理拡充工数: 中

発行済み招待コードの一覧・失効管理と、BAN・権限付与・設定変更・一斉通知の操作ログ記録を導入。以降、成約管理（4.20）・規約管理・企業審査へ拡張。
